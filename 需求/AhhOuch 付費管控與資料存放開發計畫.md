# AhhOuch 付費管控與資料存放開發計畫

> 本文件依據 [`討論資料/付費管控與資料存放建議.md`](./討論資料/付費管控與資料存放建議.md) 與
> [`AhhOuch 多人使用需求.md`](./AhhOuch%20多人使用需求.md) 撰寫，作為「轉型為消費者付費版」的開發規劃。
>
> 核心決策（已於建議文件定案）：
> - **資料存放**：本機優先（Local-First），業務資料與圖片全存手機沙盒 SQLite，開發方不經手不持有。
> - **跨裝置／備份**：交由 Apple iCloud / Google Drive 原生備份，不自建同步伺服器。
> - **付費管控**：群組分發 APK ＋ 第三方電子支付 ＋ 人工發放序號 ＋ **Email OTP 綁定**。
> - **雙平台**：前端採 **PWA**，iOS 透過 Safari「加入主畫面」、Android 另可包 APK。
> - **後端**：僅建置**極輕量序號驗證伺服器**，絕不儲存任何使用者業務資料／個資／圖片。
> - **專案拆分**：客人與店家拆成**兩套獨立程式**（各自獨立資料夾＋獨立 git repo）。
>   現有 repo 留作**店家系統**（全功能、非營利、維持桌面 exe）；**客人系統另開新 repo**，即本計畫對象。
> - **執行順序**：先清全域樣式 → 拆分為兩套 repo → 再分別做手機版打包與介面測試／修改（見 §7）。
>
> 本文件為**開發前的規劃文件**；開發過程的調整與修訂歷程改記錄於
> [`AhhOuch 開發報告.md`](./AhhOuch%20開發報告.md)。
>
> 文件版本：v1.1 ｜ 建立日期：2026-06-23（v1.1 增列專案拆分策略與執行順序）

| 使用情境 | **多人、各自獨立**；針對消費者（客人系統），手機優先 |

### 與現有版本的關係

現況見 [`AhhOuch 開發報告.md`](./AhhOuch%20開發報告.md)：現有版本為 **v1.7.0**，是「店家＋客人」雙角色的單機本機伺服器網頁軟體（**Python FastAPI + Vue 3 + SQLite**，PyInstaller 打包**單一免安裝 exe**，桌面優先、手機用響應式 RWD），客人與店家模組 **M0–M8 全部完成並持續精修**。

本計畫**把客人系統拆出，獨立為消費者付費 App**：沿用既有客人模組（養身館／看板／卡片／心得／模板）的資料模型與業務邏輯，改造為「手機 PWA ＋ 本機儲存 ＋ 免費版限制 ＋ 序號授權」的型態。店家系統則拆為**另一套獨立、全功能、非營利**的程式（維持現行桌面 exe 架構），不在本計畫里程碑內、自行演化。

> 注意：現行架構是「FastAPI 後端 + 本機伺服器 + 集中式 SQLite 檔」；本計畫的目標型態是「**無中央伺服器、資料各自存在每支手機**」。兩者的資料層思路相反，**前端技術路線與本機儲存方式（§6.1）是本計畫最大的待決與工程量來源**，並非單純沿用現有 exe。

### 專案拆分策略（兩套獨立程式）

最終交付為**兩套各自獨立的程式**，採**獨立資料夾＋獨立 git repo**（不共用、不採 monorepo）。理由：兩套拆分後在平台（桌面 exe vs 手機 PWA）、後端（本機伺服器 vs 無中央伺服器 Local-First）、儲存（集中式 SQLite vs 各手機本機）、打包（PyInstaller vs PWA/APK）、商業模式（非營利 vs 免費＋付費）全面分歧，可共用的程式碼快速趨近於零；獨立 repo 可各自 `app.toml`／版次／打包輸出／git 歷史，邊界由資料夾實體強制，對單人非程式開發者最直覺。

| 項目 | 店家系統 | 客人系統（本計畫） |
| --- | --- | --- |
| repo 來源 | **沿用現有 repo**（架構離現況最近，改動最小） | **另開新資料夾＋新 git repo**（複製現有碼後剝除店家部分） |
| 定位 | 全功能、**非營利**、桌面 exe | 消費者、**免費＋序號付費**、手機 PWA |
| 架構 | 維持 FastAPI＋Vue＋本機伺服器＋集中式 SQLite | 重構為 Local-First（無中央伺服器、各手機本機儲存） |
| 設定／版次 | 自己的 `app.toml`、打包輸出 | 自己的 `app.toml`、打包輸出 |

- 拆分作法：店家 repo 剝除客人路由／模型／view；客人新 repo 由現有碼複製後剝除店家部分，再依本計畫做 Local-First／PWA 重構。
- 相關需求文件（客人需求、多人使用需求、本計畫）複製一份至客人新 repo；現有 `需求/` 完整歷史留在店家 repo。
- **拆分後兩套各自演化，不維護「共用套件」**；唯一在拆分前先處理的共用項是全域樣式（見 §7 P0）。

### 核心架構流程

```
┌─────────────────────────────────────────────┐
│   消費者手機：AhhOuch PWA（iOS / Android）      │
│   ┌───────────────────────────────────────┐ │
│   │  前端 UI（客人系統）                       │ │
│   │  本機 SQLite（業務資料 + 圖片，沙盒儲存）    │ │
│   │  免費／付費 權限閘門（Feature Gate）         │ │
│   │  授權狀態快取（序號 + 綁定 Email）           │ │
│   └───────────────────────────────────────┘ │
└───────────────┬─────────────────────────────┘
                │ 僅「授權／換機」時連線（HTTPS）
                │ 不傳輸任何業務資料／圖片
┌───────────────┴─────────────────────────────┐
│   極輕量序號驗證伺服器                          │
│   DB：[序號, 狀態(未使用/已啟用), 綁定 Email]     │
│   Email OTP 寄送（SendGrid / Mailgun 免費方案）  │
└─────────────────────────────────────────────┘
                │
        ┌───────┴────────┐
        │ 第三方金流（人工對帳）│
        │ ECPay / LINE Pay │
        └────────────────┘
```

> 重點：手機端是完整的本機應用；伺服器**只做授權**，業務資料永不上雲，從物理層面規避敏感內容審查與封鎖風險，且伺服器近乎零維運成本。

## 3. 架構決策摘要（為何如此選）

> 出自建議文件「二、方案評估與討論過程」，此處摘要供開發對照。

| 決策面 | 採用方案 | 捨棄方案與原因 |
| --- | --- | --- |
| 資料存放 | **本機優先（Local-First）** | 捨「雲端集中儲存」：圖片恐涉敏感內容易違反雲端 AUP 遭封鎖，且流量／儲存成本隨人數遞增 |
| 跨裝置／備份 | iCloud / Google Drive 原生備份 | 不自建同步伺服器，責任交還使用者 |
| 付費管控 | **群組分發 ＋ 電子支付 ＋ 人工序號** | 捨 App Store/Google Play IAP（抽成 15–30%、成人內容難過審）；捨「網頁付費＋帳號系統」（違背免帳號輕量初衷） |
| iOS 分發 | **PWA 加入主畫面** | iOS 無法直接安裝 APK |
| 金流 | **第三方電子支付 / LINE Pay** | 捨實體網銀轉帳：易遭詐騙集團利用導致帳戶被列警示戶凍結 |
| 授權防盜版 | **Email 綁定 ＋ Email OTP** | 捨純序號（易群組共用）、捨硬體 ID 綁定（換機需人工客服）、捨 SMS OTP（每封約 1 元成本）。Email OTP 近乎零成本，且消費者不會把信箱密碼給人 → 天然防共用 ＋ 全自動換機 |

## 4. 系統架構與模組設計

### 4.1 前端 App（PWA，客人系統）
- 沿用既有客人模組功能：養身館、看板（Trello 式）、美容師卡片、簡介（圖片／國籍／文字）、心得（日曆／評分模板／多組）。
- **PWA 化**：加入 `manifest.json`（名稱、圖示、`display: standalone`、主題色）與 Service Worker（離線快取、可安裝）。iOS 經 Safari「加入主畫面」。
- **本機資料層**：資料與圖片改存手機端（見 §4.2）。
- **Feature Gate（權限閘門）**：依授權狀態（免費／付費）控制可建立的養身館卡片數與簡介圖片數（F1/F2/F5）；觸發上限時跳出升級引導（F4）。

### 4.2 本機資料層（Local-First，取代伺服器 SQLite）
- 業務資料庫與圖片皆存於手機端沙盒：
  - **PWA 純網頁路線**：以 IndexedDB（或 SQLite WASM + OPFS）存資料，圖片以 Blob 存 IndexedDB。
  - **Flutter / React Native 路線**：直接用裝置 SQLite ＋ 檔案系統沙盒（建議文件提及 `sqflite` 類方案）。
- 設計原則：**App 完全離線可用**；伺服器僅在「授權／換機」時短暫連線，且不傳業務資料（B3/D1/D2）。

> 技術路線（PWA 純網頁 vs Flutter/RN）為本計畫**首要 PoC**，見 §6.1。

### 4.3 授權模組（前端）
- 狀態機：`未授權(免費) → 開通中(等待 OTP) → 已授權(付費)`。
- 開通：輸入序號 ＋ Email → 呼叫後端 `request-otp` → 輸入 6 碼 → 呼叫 `verify-otp` → 成功則本機寫入授權快取（L1/L2/L3）。
- 換機：選「已有授權」→ 只輸入原 Email → 收 OTP → 驗證成功即解鎖（Q1/Q2）。
- 授權快取存於本機（如簽章後的 token / 旗標）；離線時以本機快取維持付費版（L3）。

### 4.4 序號驗證伺服器（後端，極輕量）
- 單一資料表：`license(serial PK, status[未使用/已啟用], bound_email, activated_at, …)`。
- API（最小集）：
  - `POST /licenses`（後台用）：產生唯一序號（如 `AHH-123`），狀態「未使用」（Y3）。
  - `POST /activate/request-otp`：收 `{serial, email}`，驗證序號可用，寄 6 碼 OTP（B2）。
  - `POST /activate/verify-otp`：收 `{serial, email, code}`，正確則將序號綁定該 Email、狀態改「已啟用」、回授權憑證（L2）。
  - `POST /migrate/request-otp` / `verify-otp`：換機用，只憑「原綁定 Email」寄碼／驗證（Q1）。
- **風控**：同一 Email 短時間於大量不同裝置請求解鎖 → 自動阻擋或告警（Q3）。
- **嚴格邊界**：伺服器只碰序號與 Email，**不存任何業務資料／圖片／個資**（B3）。
- 寄信：SendGrid／Mailgun 免費方案（B2）。

### 4.5 後台管理（管理員）
- 確認金流款項後，產生並查詢序號、查看綁定狀態（Y3）。初期可極簡（受保護的後台頁或 CLI 腳本）。

### 4.6 營運與金流（非程式，SOP）
- 收款一律走第三方電子支付（ECPay 虛擬帳號／超商代碼 或 LINE Pay），**不提供實體網銀帳號**（Y1/Y2）。
- 完整 SOP 見 §8。

## 5. 資料結構設計（初稿）

### 手機端（本機，業務資料）
```
（沿用既有客人模組，存於手機沙盒）
養身館(Spa)：id / 名稱* / 地址 / 幹部 / 看板列表

評分模板(RatingTemplate)：id / 名稱 / 項目清單

授權快取(LicenseCache，本機)：序號 / 綁定Email / 狀態 / 授權憑證 / 最後驗證時間
```

### 伺服器端（僅授權，最小化）
```
license：
  serial*（唯一序號，如 AHH-123）
  status（未使用 / 已啟用）
  bound_email（綁定信箱，啟用後填入）
  activated_at / last_migrate_at
otp（暫存，可設 TTL）：
  email / serial / code（6 碼）/ expires_at / attempts
```

> 伺服器資料表刻意極小，符合 B1/B3「只做授權、不持有業務資料」。

## 6. PoC / 待研究項目

> 不阻擋整體規劃，但需在對應階段前先驗證並選定方案。

### 6.1 前端技術路線選型 —【最優先，M1 前定案】
- **路線 A：PWA 純網頁**（Vue 沿用既有前端 + Service Worker + IndexedDB/SQLite-WASM）
  - 優點：最大程度沿用現有 Vue 客人模組、單一程式碼、iOS/Android 通吃。
  - 待驗證：iOS Safari PWA 的本機儲存穩定度、相機／剪貼簿貼圖、加入主畫面後的儲存持久性、無 APK 也能用。
- **路線 B：Flutter / React Native**（匯出 PWA ＋ Android APK）
  - 優點：原生 SQLite 與檔案沙盒、效能與裝置整合佳。
  - 待驗證：需重寫前端、學習成本、與既有 Vue 程式碼落差。
- 初步傾向 **路線 A（PWA）** 以沿用既有資產；若本機儲存或 iOS 限制踩雷再評估 B。

### 6.2 本機儲存與離線可用性 — M1 前驗證
- 驗證所選路線下：圖片（多張）存取、容量上限、清除快取是否誤刪資料、PWA 離線啟動。

### 6.3 iCloud / Google Drive 原生備份 — M7 前驗證
- PWA 路線下本機資料是否能被原生備份涵蓋；必要時提供「匯出／匯入備份檔」作為保底（見 §10 待確認）。

### 6.4 第三方金流串接程度 — M5 前確認
- ECPay 虛擬帳號／超商代碼是否需 API 對帳，或人工對帳即可；LINE Pay 收款流程。

### 6.5 Email OTP 寄送 — M4 前驗證
- SendGrid／Mailgun 免費額度、寄達率、退信處理、驗證碼 TTL 與重試上限。

## 7. 開發階段與里程碑

> **執行順序原則（先拆分、再分別修介面）**：拆分定義了「要打包什麼、底層長怎樣」，手機打包是建在拆分結果上的；

## 8. 營運與自動化換機標準作業流程（SOP）

> 出自建議文件「三、3」，整理為可執行步驟。

### 購買與首次開通
1. 消費者透過群組的第三方金流連結完成付款。
2. 管理員確認款項後，於後台產生唯一序號（如 `AHH-123`）發給消費者。
3. 消費者在 App 內輸入「序號」＋「個人 Email」。
4. 系統寄 6 碼驗證碼至該 Email；輸入正確 → 序號綁定此 Email、App 解鎖為付費版。

### 全自動換機（無須客服）
1. 消費者於新手機下載安裝 App。
2. App 內點「已有授權」，輸入**原綁定 Email**。
3. 系統再寄一封驗證碼至該信箱。
4. 輸入正確 → 新手機即刻解鎖為付費版。
5. 防盜版：伺服器偵測同一 Email 短時間於大量不同裝置請求解鎖 → 自動阻擋或告警。

## 9. 風險與注意事項

- **iOS PWA 儲存持久性**：iOS Safari 對網頁本機儲存有清除策略，長期不用或清快取可能遺失資料；需在 §6.2 驗證並以「備份」保底（D3）。
- **敏感內容審查**：本機優先已從物理層規避雲端封鎖；務必確保伺服器與金流端**完全不經手圖片／業務資料**，維持低風險定位（B3）。
- **金流帳戶安全**：嚴禁使用實體網銀帳號收款，避免被列警示帳戶凍結（Y2）；一律走電子支付／LINE Pay。
- **序號共用防護**：Email OTP 為主要防線；仍需後端風控（同 Email 多機解鎖偵測）避免大規模盜用（Q3）。
- **寄信額度與退信**：免費寄信方案有額度與退信風險，需監控寄達率、設 OTP TTL 與重試上限（B2）。
- **既有程式碼落差**：若選 Flutter/RN 路線，現有 Vue 客人模組需重寫，工期與學習成本大增；選型須謹慎（§6.1）。
- **App 商店政策**：採群組分發／PWA 以規避上架審查；須留意 iOS 側 PWA 能力與 Android 側「未知來源安裝」的使用者引導。
