# AhhOuch 資料存放與系統架構建議 (含風險對策)

> 本文件基於 AhhOuch 開發計畫，針對不同人數規模、隱私需求與資安挑戰，提出三種層級的架構方案，並詳細列出各類潛在風險與防禦對策，方便您全面評估。

---

## 1. 依據人數級距的架構方案建議

### 方案 A：極小規模 (1~10 人內) —— 「虛擬區域網路」伺服器
* **適用情境**：僅限開發者與少數親友使用，想省下主機與維護成本。
* **資料存放**：開發者的個人筆電硬碟（SQLite 單一資料庫 + 本機 `/DATA/images` 資料夾）。
* **網路架構**：**絕對不開放**家用路由器的公網 Port Forwarding。改用 **Tailscale 或 ZeroTier** 建立私密虛擬區域網路 (VPN)，僅允許安裝該軟體的 10 位使用者連線至筆電的內部 IP。
* **潛在風險與應對方式摘要**：
  * **駭客資安風險**：依賴 Tailscale 阻絕公網掃描。
  * **伺服器營運風險**：需設定筆電防休眠，並手動備份 `DATA` 資料夾防硬碟損毀。
  * **情色與法律風險**：因實體持有圖檔，需約束這 10 位使用者的上傳內容。

### 方案 B：中大型規模 (50~500 人以上) —— 雲端伺服器 + 雲端物件儲存
* **適用情境**：公開給大量外部客人使用，且多數使用者透過手機行動網路存取。
* **資料存放**：文字資料庫升級為 **PostgreSQL**，圖片檔案改存於 **雲端物件儲存空間 (如 AWS S3, Cloudflare R2)**，搭配 CDN 加速以防止塞爆單一主機頻寬。
* **網路架構**：FastAPI 部署於雲端 VPS (如 Linode, EC2)，Vue 3 前端獨立託管於 Vercel 或 Cloudflare Pages 達到秒開體驗。
* **潛在風險與應對方式摘要**：
  * **情色圖片審查風險**：選擇對合法成人內容寬容的底層儲存 (如 AWS S3) 避開雲端 AI 封鎖。
  * **駭客攻擊風險**：外部套用 **Cloudflare WAF**，並落實 API 權限控管與檔案格式檢查。
  * **系統崩潰與資料遺失**：設定 VPS 每日自動快照，確保災難還原能力。

### 方案 C：極致隱私保護方案 —— 「本機優先 (Local-First)」架構
* **適用情境**：使用者極度重視隱私，預期有大量敏感/成人圖片，強烈抗拒資料上傳至他人伺服器。
* **資料存放**：**全部存放在使用者的個人手機/電腦裝置內**（如網頁 IndexedDB，或打包成手機 App 存入本機 SQLite）。
* **網路架構**：雲端伺服器退居幕後，只負責派發網頁程式碼，不儲存任何實質的心得與圖片。
* **潛在風險與應對方式摘要**：
  * **情色內容持有風險**：圖片不離開客人的手機，從源頭完美解決。
  * **備份檔審查風險**：實作端對端加密備份 (AES-256)，讓客人安心上傳自己的雲端空間。
  * **忘記密碼風險**：強制設定密碼提示，加上強烈的免責聲明防呆機制。

## 2. 各種潛在風險的詳細說明與應對方式

### 2.1 情色與敏感內容風險
* **風險描述**：使用者高機率上傳具有裸露性質的照片。若存於主流雲端代管平台，極可能因違反「合理使用政策 (AUP)」遭 AI 掃描並永久封鎖帳號；若存於開發者筆電，開發者將背負實體持有爭議（甚至非法內容）的法律風險。
* **對應策略**：
  * **(最佳解) 採用方案 C (本機優先)**：圖片永遠不離開客人的手機，從源頭解決雲端審查與開發者持有問題。
  * **端對端加密備份 (E2EE)**：若客人需把資料備份到自己的 Google Drive/iCloud，系統應強制使用者設定「備份密碼」，將圖片庫打包成 **高度加密的亂碼檔案 (如 AES-256 .enc)**。雲端 AI 無法掃描加密檔，保護客人自己的雲端帳號免受封鎖。
  * **挑選基礎設施**：若走方案 B，應避開審查嚴格的平台（如 Vercel, Netlify 託管圖檔），選擇對成人內容較寬容的 AWS S3 或是無干預的基礎設施。

### 2.2 駭客與資訊安全風險
* **風險描述**：伺服器連上網際網路必定面臨全世界駭客的無差別掃描；若軟體出現漏洞，可能導致全部 500 人的資料外洩。
* **對應策略**：
  * **災情隔離**：絕對避免開放個人筆電外網。必須上線公網時應使用雲端 VPS（方案 B），即使被駭客破壞，也能將風險隔離在該台雲端主機，不牽連開發者私領域檔案。
  * **加上防護盾**：在雲端主機外部套用 **Cloudflare WAF (網站應用程式防火牆)**，可阻擋 99% 的惡意機器人與 DDoS 攻擊。
  * **軟體層防禦**：必須落實 API 權限控管 (JWT Token)，並**嚴格檢查上傳檔案的格式**，防止駭客上傳惡意腳本執行檔。

### 2.3 伺服器營運與硬體風險
* **風險描述**：使用個人筆電當伺服器時，會面臨休眠、斷線、硬體損毀（例如硬碟壞軌）導致所有人無法使用且資料歸零的風險。
* **對應策略**：
  * 若人數到達一定規模，應果斷轉移至雲端 VPS。雲端機房具備不斷電系統，並應設定每日自動「主機快照 (Snapshot)」，一旦遭遇崩潰，可於幾分鐘內一鍵還原至昨天的狀態。

### 2.4 其他衍生風險 (一)：使用者忘記密碼與資料遺失
* **風險描述**：在「本機優先」或「加密備份」架構下（零知識加密），開發者並未持有解密金鑰。一旦使用者手機遺失、清除快取，且**忘記了自己設定的備份密碼**，資料將遭受物理性的永久毀滅，無法救援。
* **對應策略**：
  * **強制密碼提示**：設定密碼時，強制要求填寫純文字的密碼提示（如：第一隻寵物的名字），能解決多數遺忘狀況。
  * **防呆警告設計**：UI 需加上極醒目的免責聲明，強制勾選「我了解若忘記密碼，任何人包含開發者都無法幫我還原，資料將永久遺失」。
  * **整合生物辨識**：在同一台裝置上，整合 FaceID 或指紋解鎖來調用金鑰，減少記憶負擔。

### 2.5 其他衍生風險 (二)：跨裝置同步困難
* **風險描述**：若採用「方案 C（本機優先）」，資料綁死在單一裝置。客人無法輕易達到「在手機上傳圖片，在電腦大螢幕觀看」的無縫切換體驗。
* **對應策略**：
  * 必須開發完善的「單鍵打包匯出 / 匯入」功能。
  * 未來擴充可研究基於 WebRTC 的 P2P 點對點連線技術，讓客人的手機與電腦在同一個 Wi-Fi 下能直接互傳同步資料，全程不經由第三方伺服器。

### 2.6 受邀使用者（內鬼）攻擊：VPN 防外不防內（採方案 A + Tailscale/ZeroTier 後的重點）

> 已決定採方案 A（微規模）並以 **Tailscale / ZeroTier 虛擬區域網路**連線。
> 本節針對「**你邀請進虛擬區網、卻心懷不軌的使用者**」這個內部威脅，依現行程式碼提出分層對策。
> 撰寫日期：2026-06-20，依 0.6.0 程式碼實查。

**核心觀念**：VPN 只做兩件事——**只讓受邀裝置連得到、且全程加密**，擋掉全網匿名掃描與攻擊。
但它對「受邀後作惡的成員」幾乎無能為力。而目前程式**無任何登入驗證、亦無使用者隔離**，因此：

> ⚠️ **VPN 內任何一個成員 ＝ 對全部資料的完整管理員權限**
> （可讀寫刪除所有人的養身館／心得，亦看得到並能觸發社群發布權杖）。

故「成員中有人攻擊」須分層應對：

#### 第 0 層 — VPN 本身已提供的防線（免費利器）
* **裝置身分**：tailnet 內每個連線都是具名的已知裝置，而非匿名 IP。
* **即時撤銷**：發現惡意成員 → 於 Tailscale 管理後台**一鍵將該裝置踢出**，立即斷線。這是對付內鬼最強的槓桿。
* **ACL 存取清單**：可限定「僅哪些成員／裝置能連到 8000 埠」，不必每個 tailnet 成員都能存取。
* **防火牆只綁 VPN 介面**：Windows 防火牆入站規則限定 Tailscale 網卡，**勿開放至家用 Wi-Fi 網段**，避免在區網裸奔。

#### 第 1 層 — 最大缺口：無 App 層驗證（建議優先補）
目前無登入 ＝ 成員彼此資料全互通、權杖可被任意成員觸發。若成員間本就該互看，可接受；
若不該，**至少加一道共用密碼**，要隔離各人資料才上帳號系統。**這是現況最大的洞，VPN 補不了。**

#### 第 2 層 — 實際「駭客手法」的攻擊面（依現行程式碼實查）

| 攻擊手法 | 現況 | 對策 |
| --- | --- | --- |
| **SQL injection** | ✅ **安全**：全用 SQLModel ORM 參數化查詢；唯一原生 SQL（`database.py` 遷移）為寫死表名，非使用者輸入 | 維持現狀，勿引入字串拼接 SQL |
| **路徑穿越（寫/讀檔）** | ✅ **安全**：上傳檔名為伺服器產生的 uuid，圖片經 StaticFiles 掛載 | 維持現狀 |
| **惡意檔案上傳** | ⚠️ 經 Pillow 重新解碼＋另存，可洗掉多數惡意 payload | 已做得不錯，維持 |
| **塞爆硬碟／記憶體 (DoS)** | ❌ **有風險**：`await file.read()`（`routers/customer.py`、`routers/store.py`）**無大小上限**，可上傳數 GB 或狂洗貼上 | **加上傳大小上限 ＋ 拒絕超標**（小改動） |
| **解壓縮炸彈** | ❌ **有風險**：`img.load()` 對構造過的小圖可膨脹成巨量像素吃光記憶體 | 設 `Image.MAX_IMAGE_PIXELS` 並 try/except |
| **SSRF（誘導伺服器打內網）** | ✅ **安全**：發布只打死的 LINE/Telegram 網址，host 非使用者可控 | 維持現狀 |
| **權杖濫用** | ⚠️ 回應已遮蔽只露末四碼，但任何成員可**觸發發送／改權杖**，且權杖**明文存 DB** | 發布功能應上驗證；考慮加密儲存權杖 |
| **無速率限制** | ⚠️ 任何端點可被狂打 | 視需要加 rate limit（次要） |

#### 第 3 層 — 縮小爆炸半徑（萬一 App 被玩壞）
伺服器跑在**開發者個人筆電、以其帳號權限**執行；若出現 RCE 級漏洞，攻擊者即取得該帳號。對策：
* 用**獨立低權限 Windows 帳號**或容器／VM 執行伺服器，使其讀不到開發者私人檔案。
* 維持 `DATA/` 為應用唯一可寫路徑（現況已是）。
* 防火牆規則僅綁 VPN 介面，限制可達範圍。

#### 第 4 層 — 偵測與復原
* **日誌**：保留 uvicorn 存取日誌（含來源 IP，可對應到 tailnet 具名裝置），事後可追查。
* **備份**：定期複製 `DATA/`（DB ＋ images），讓惡意刪除可還原——目前**尚無備份**（見開發報告 §9-2），
  這是對付「內鬼清空資料」CP 值最高的保險。
* **事件處置**：於 Tailscale 後台撤銷該裝置即刻斷其連線。

#### 優先順序建議（1~10 人 VPN 情境）
1. **Tailscale ACL ＋ 防火牆只綁 VPN 介面**：限制誰／哪台裝置能連到 8000，並隨時可撤銷。
2. **補 App 層驗證**（至少共用密碼）：填補「VPN 成員＝全權管理員」這個最大缺口。
3. **強化上傳**：大小上限 ＋ 解壓縮炸彈防護（小改動，擋住磁碟／記憶體 DoS）。
4. **備份 `DATA/`**：可從惡意刪除中復原。
5. **執行環境收斂**：以低權限帳號執行、防火牆綁 VPN 介面。
6. （速率限制：有餘力再加。）
