# 電腦版／手機版 資料同步與連線方案評估

> **本文件用途**：在「讓手機版能夠使用」之前，先釐清電腦版與手機版之間「資料怎麼同步」與
> 「手機怎麼連到伺服器」兩件事，列出可行方案、評估原因與優缺點，供拍板決策。
>
> 撰寫日期：2026-06-20 ｜ 對應版本：0.6.0
> 參考但不沿用：[`資料存放建議.md`](./資料存放建議.md)（該文是「整體架構選型 A/B/C」，本文聚焦現階段落地）
>
> **⚠️ 前提變更（2026-06-20 補）**：使用者後續調整方向——**以「放棄開發者電腦當主要伺服器」為前提**，
> 優先序改為 **避免敏感內容審查 ＞ 防駭客攻擊 ＞ 使用者隱私 ＞（操作方便性後置）**。
> 在此新優先序下，原被列為「不採用」的 **方案 D（本機優先）上修為主要候選**。
> §0／§1 的「無需同步、D 反效果」是**舊優先序（方便性優先）下**的結論，保留供對照；
> 方案 D 的完整評估見下方 §2 方案 D 與新增的 **§6（本機優先深入設計）**。

## 0. 一句話結論（先看這個）

**現有架構沒有「兩份資料要同步」的問題。** 全系統只有一台伺服器、一份 SQLite ＋一個圖片資料夾；
電腦與手機都只是**瀏覽器（瘦客戶端）**，看的是同一份資料 → **天生即時同步、零衝突**。
要解的其實是「**手機如何連到這台電腦的伺服器**」這個網路問題。

---

## 1. 為什麼「不需要做同步機制」（先釐清前提）

驗證自現行程式碼，三個事實決定了這個結論：

| 事實 | 來源 | 含義 |
| --- | --- | --- |
| 前端 API 全用**相對路徑**（`/api/...`、`/images/...`） | `frontend/desktop/src/api.js` | 呼叫的永遠是「送出此網頁的那台伺服器」，換裝置不需改設定 |
| 手機版＝**響應式同一套前端**（非獨立 app），由 FastAPI 直接提供 | 開發報告 §2、M7 | 手機不另存資料、不需各自的資料庫 |
| 資料只有一份：SQLite ＋ `DATA/images/`，在電腦上 | 開發報告 §3 | 沒有第二份副本，也就沒有「合併／衝突／延遲」問題 |

> 換句話說：手機上傳一張圖、電腦重新整理就看得到，因為**兩邊本來就在讀寫同一個檔案**。
> 這是「單一事實來源（single source of truth）」架構自帶的好處，無需額外開發。

⚠️ **此結論的適用範圍**：上述「天生即時同步、無需同步機制」**只在維持「單一伺服器」架構（方案 A/B/C）時成立**。
若改採**方案 D（本機優先 / 各裝置各存一份）**，就會**主動製造**跨裝置同步難題（「資料存放建議.md」§2.5
所述「資料綁死單一裝置」）——這正是方案 D 用「即時同步的方便性」**換取**「免審查、防駭、隱私」的代價。
在本文件頂端「前提變更」的新優先序下，**這個代價是可接受的取捨**，其處理方式見 §2 方案 D 與 §6。

## 2. 真正要解的問題：手機如何連到伺服器

目前伺服器綁定 `127.0.0.1:8000`（`app.toml`），只接受**本機**連線，手機連不到。
以下是讓手機連得到的可行方案，由輕到重排列。

### 方案 A — 同一 Wi-Fi 區域網路（區網直連）

**做法**
1. 伺服器綁定改為 `0.0.0.0:8000`（監聽所有網卡，`app.toml` 一行）。
2. Windows 防火牆放行 TCP 8000 入站。
3. 手機連同一個 Wi-Fi，瀏覽器開 `http://<電腦區網IP>:8000`（本機目前為 `192.168.213.127`）。

**評估原因**：對應「資料存放建議.md」**方案 A（1~10 人微規模）**，是目前人數與隱私需求下成本最低、
最快可用的解，且程式幾乎不用改（前端相對路徑、`run.py` 已處理 `0.0.0.0` 時用 localhost 開瀏覽器）。

| 優點 | 缺點 |
| --- | --- |
| 零額外軟體、免費 | 手機與電腦**必須在同一 Wi-Fi**，外出行動網路連不到 |
| 資料完全不出家門，隱私最佳 | 區網 IP 由 DHCP 配發**可能變動**（可在路由器設固定 IP/保留位址解決） |
| 延遲最低（區網內幾乎即時） | 同網段任何裝置都連得到 → 需考量存取控制（見 §3） |
| 改動最小、風險可控 | 需手動輸入 IP（可用 QR code / 固定 IP 緩解） |

### 方案 B — Tailscale / ZeroTier 私密 VPN

**做法**：電腦與手機各裝 Tailscale（或 ZeroTier），登入同一帳號形成私密虛擬區網；
手機用分配到的虛擬 IP（如 `100.x.x.x`）連 `http://<虛擬IP>:8000`，**不開任何公網 Port**。

**評估原因**：當需求從「在家用」擴大到「外出用行動網路也要連」時，這是「資料存放建議.md」
**方案 A 網路架構**明確推薦的做法（取代危險的家用路由器 Port Forwarding）。

| 優點 | 缺點 |
| --- | --- |
| 手機在**任何網路**（行動數據／別人家 Wi-Fi）都能連 | 電腦與手機各需安裝一個 App 並登入帳號 |
| 流量端對端加密，**不暴露公網**、不被全網掃描 | 依賴第三方協調伺服器（僅協調連線，不經手資料） |
| 虛擬 IP 固定，不隨 DHCP 變動 | 多一層設定與帳號維護成本 |
| 與方案 A 可並用（在家走區網、外出走 VPN） | 免費方案有裝置數上限（個人用通常足夠） |

### 方案 C — 雲端伺服器（VPS + 物件儲存）

**做法**：把後端搬上雲端 VPS、圖片改存 S3/R2、前端託管 Vercel/Cloudflare Pages。

**評估原因**：對應「資料存放建議.md」**方案 B（50~500 人）**。**現階段過頭**，
且把敏感圖片放上第三方雲端會引入內容審查與帳號封鎖風險（該文 §2.1）。**本階段不採用**，
僅列為未來人數放大時的升級路徑。

| 優點 | 缺點 |
| --- | --- |
| 不依賴開發者電腦開機、可大量併發 | 月租成本、維運複雜度高 |
| 公開可及、體驗佳 | 敏感圖片上雲有審查／封鎖／法律持有風險 |
| — | 需重做部署、儲存、資安（WAF、JWT）整套 |

### 方案 D — 本機優先（各裝置各存一份）　★新優先序下的主要候選

**前提**：以「**放棄開發者電腦當主要伺服器**」為前提，優先序為
**避免敏感內容審查 ＞ 防駭客攻擊 ＞ 使用者隱私 ＞（方便性後置）**。對應「資料存放建議.md」**方案 C**。

**做法（概念）**：資料（DB ＋ 圖片）存於**各使用者自己的裝置**，無常態伺服器。裝置間以
「**單鍵加密匯出 → 任意通道傳檔 → 另一裝置匯入合併**」交換資料，合併時依規則處置衝突
（保留 A／保留 B／兩者皆保留）。**完整設計見 §6。**

**評估原因**：本機優先讓敏感圖片**永不以明文離開裝置**，從源頭消滅雲端／LINE 的 AI 內容審查；
無常態伺服器可被駭、攻擊面極小；開發者亦不持有任何使用者資料（去除持有爭議與法律風險）——
精準命中新優先序的前三項。代價是跨裝置不再即時同步、需手動匯出匯入並處理衝突（方便性後置，符合新取捨）。

| 優點 | 缺點 |
| --- | --- |
| 圖片**永不明文離開裝置** → 從源頭免雲端/LINE 內容審查 | 無即時同步，需手動匯出／匯入 |
| 無常態伺服器 → 攻擊面極小、無中央資料庫可一次外洩 | 須自建合併與**衝突處置**（本方案最難部分，§6.4） |
| 開發者不持有任何使用者資料 → 去除持有爭議與法律風險 | **架構大改**：資料層需從伺服器搬到裝置端、ID 需改 UUID（§6.1–6.2） |
| 可離線使用、不依賴開發者電腦開機 | 忘記備份密碼＝資料永久遺失（需密碼提示＋免責，§6.5） |

## 3. 連帶決策：存取控制（綁 0.0.0.0 後才浮現）

一旦改綁 `0.0.0.0`（方案 A/B 都需要），**同網段／同 VPN 內任何裝置不需登入即可存取**
（目前系統無任何驗證機制）。需決定如何處理：

| 選項 | 說明 | 適用 | 代價 |
| --- | --- | --- | --- |
| **不另加驗證** | 倚賴 Wi-Fi／VPN 本身把外人隔絕在外 | 1~10 位信任親友、家用網路 | 最快可用；但同網段他人可存取 |
| **加一道簡單密碼／Token** | 後端加最基本的存取碼，首次連線需輸入 | 想多一層保險 | 需新增後端驗證 ＋ 前端登入流程 |
| **完整帳號系統（JWT）** | 多使用者各自帳密與權限 | 規模放大、對外開放時 | 開發量大，現階段過頭 |

> 「資料存放建議.md」§2.2 建議公網上線才需 JWT；區網／VPN 內的 1~10 人情境，
> 「不另加驗證」即符合方案 A 的原始設計，但**這是安全取捨，需由使用者拍板**。

## 4. 建議（待確認，非定案）

本文件存在**兩條互斥方向**，取決於優先序：

**方向一 — 方便性優先（維持單一伺服器）**：先落地**方案 A（區網直連）**，外出再疊**方案 B（Tailscale）**。
天生即時同步、改動最小，但資料集中於開發者電腦、需處理存取控制（§3）。

**方向二 — 隱私／免審查優先（新優先序，使用者選定方向）**：採**方案 D（本機優先）**。
資料各存裝置、加密匯出匯入、自行處理衝突。從源頭免審查、防駭、保護隱私，**代價是架構大改與方便性下降**（§6）。

> 取捨關鍵一句：**要「即時同步的方便」就走 A/B；要「免審查＋防駭＋隱私」就走 D**，兩者難以兼得。
> 不採用方案 C（雲端，過頭且敏感圖片上雲有審查風險）。

### 若採方案 A（方向一），預期實作項目（供估量，未動工）

- `app.toml`：`host` 由 `127.0.0.1` 改 `0.0.0.0`（影響伺服器對區網開放，屬對外變更）。
- Windows 防火牆：新增 TCP 8000 入站規則。
- 便利性：啟動時於終端印出手機可開的區網網址，並可選擇產生 QR code 供手機掃描直連。
- 驗證：以實機手機連同網段測試響應式介面與圖片上傳同步。

### 若採方案 D（方向二），預期實作項目（供估量，未動工）

- **架構**：選定 D1（PWA+IndexedDB）或 D2（原生打包+本機 SQLite）；資料層由伺服器搬至裝置端（§6.1）。
- **ID 改造**：主鍵由自增整數改 **UUID**，並為各表加「最後修改時間」與「刪除墓碑」欄位（§6.2、§6.4）。
- **匯出／匯入**：單鍵 AES-256 加密匯出整份資料（manifest＋圖片）為可攜檔，及對應匯入（§6.3）。
- **衝突處置**：實作「保留 A／保留 B／兩者皆保留」合併引擎（§6.4）。
- **防遺失**：密碼提示、免責勾選、（同裝置）生物辨識解鎖金鑰（§6.5）。

## 6. 方案 D（本機優先）深入設計

> 本節依「前提變更」的新優先序，把方案 D 從概念落到可估量的設計：架構選擇、識別改造、
> 匯出匯入與傳輸通道、衝突處置、防遺失。**仍屬評估，未動任何程式。**

### 6.1 架構選擇：資料存在裝置的哪裡

現況：商業邏輯與資料都在 Python 後端（FastAPI＋SQLite），前端只是瀏覽器。本機優先的關鍵難題是
**「手機上沒有 Python 可跑」**，故資料層與邏輯必須搬到裝置端。兩條可行路線：

| 路線 | 做法 | 優點 | 缺點 |
| --- | --- | --- | --- |
| **D1：PWA ＋ 瀏覽器儲存（IndexedDB）** | Vue 改純前端 App，資料與圖片(Blob)存 IndexedDB；可安裝成 PWA、離線可用 | 不需各平台打包、沿用現有前端最多 | 瀏覽器儲存**可能被清除**/有配額上限；大量大圖較吃力；邏輯需從 Python 移植到 JS |
| **D2：原生／容器打包 ＋ 本機 SQLite** | 桌面 Electron、手機 Capacitor，各自內嵌 SQLite＋檔案系統存圖 | 儲存穩固、容量大、可整合生物辨識解鎖 | 需維護**雙平台打包**；脫離現有「單一 exe＋響應式」模式，工程量最大 |

> ⚠️ 不論 D1/D2，**現有 FastAPI/SQLModel 資料層大多需被裝置端資料層取代**：`api.js` 對伺服器的呼叫
> 改為對本機資料存取；自然排序、模板同步、班表換算等邏輯需在裝置端重實作。**這是方案 D 最大的工程成本。**

### 6.2 跨裝置識別：必須改用全域唯一 ID（UUID）

現況卡片／心得等以**自增整數**為主鍵——多裝置下會無意義碰撞（手機的 #5 ≠ 電腦的 #5），
無法判斷「兩筆是不是同一筆」。本機優先**必須**在建立資料時就產生 **UUID** 當主鍵，
讓同一邏輯記錄在所有裝置上有相同 ID，合併才能逐筆比對。**這是方案 D 的地基，需先改造。**

### 6.3 匯出／匯入與傳輸通道

**匯出**：單鍵把整份資料（DB 內容序列化為 JSON manifest ＋ 全部圖片）打包成一個可攜檔，
**建議以 AES-256 加密成亂碼檔（如 `.ahhouch`）**——加密後任何通道（含雲端／LINE）都只看到密文、
無法掃描內容，這正是達成「免審查」目標的關鍵。

**傳輸通道**（皆只是「搬一個加密檔」的啞管，互不互斥，使用者依情境挑）：

| 通道 | 說明 | 注意 |
| --- | --- | --- |
| USB 傳輸線 / 隨身碟 / SD 卡 | 裝置間直接複製檔案 | 最私密、無第三方；手機需檔案管理權限 |
| LINE／即時通訊傳檔 | 把加密檔當檔案訊息傳 | **務必加密**否則 LINE 可掃描；受其檔案大小上限約束 |
| 區網（沿用方案 A 概念） | 同 Wi-Fi 下臨時起一個下載點互傳，**非常態伺服器** | 僅傳輸期間開啟、傳完關閉，攻擊面小 |
| 雲端硬碟當啞管（Drive/iCloud/Dropbox） | 上傳**加密**檔，雲端只存密文 | 對應「資料存放建議.md」§2.1 的 E2EE 備份；雲端無法審查密文 |
| 系統就近分享（AirDrop／Nearby Share／藍牙） | OS 級裝置對裝置 | 免網路、適合同場域 |
| 電子郵件附件 | 寄加密檔 | 受附件大小上限 |
| WebRTC P2P（進階） | 兩裝置直連互傳，不經第三方儲存 | 對應「資料存放建議.md」§2.5；需協調伺服器牽線 |
| QR code（僅限小資料／金鑰） | 掃碼傳遞 | 圖片太大不適用，僅適合傳密碼/設定 |

### 6.4 資料衝突處置（合併規則）

每台裝置各自獨立修改、再互相匯入，合併是本方案**最難**的部分。設計如下：

**前提**：每筆記錄有 UUID（§6.2）＋「最後修改時間」欄位。匯入時逐筆比對：

- **只存在於匯入檔** → 直接新增。
- **兩邊都有且內容相同** → 略過。
- **兩邊都有且內容不同** → 視為**衝突**，套用使用者選的策略：

| 策略 | 行為 | 適用 |
| --- | --- | --- |
| **保留 A（本機優先）** | 維持本機現有，忽略匯入版本 | 信任本機這台是主力 |
| **保留 B（匯入優先）** | 以匯入版本覆蓋本機 | 信任來源那台較新 |
| **兩者皆保留**（你指定的預設） | 匯入版本以**新 UUID 複製成一筆新資料**並列存在，由使用者事後自行檢視、編輯或刪除 | **不丟任何資料**，最安全，符合隱私／防遺失優先 |

**粒度建議**：以「**整筆實體**」為衝突單位（例如整張卡片、整則心得），而非欄位級，較單純可控；
卡片底下的圖片／評分隨該卡片整包處理，「兩者皆保留」即整包複製一份新的。

**刪除的難題（須留意）**：單純合併會「**復活**」已刪資料——若某筆在 A 刪除、在 B 修改，
naive 合併會把它救回來。正解是**刪除時留下「墓碑（tombstone）」標記**並一起傳遞合併，
合併方看到墓碑才知道該筆是「被刪」而非「還沒同步到」。此為正確刪除合併的必要機制，需納入設計。

### 6.5 忘記密碼／資料遺失（沿用「資料存放建議.md」§2.4）

加密匯出後，開發者不持金鑰；使用者一旦忘記備份密碼且裝置遺失，資料**永久無法救回**。對策：
強制填寫純文字密碼提示、UI 醒目免責聲明（勾選「忘記密碼無人能還原」）、同裝置可整合 FaceID／指紋解鎖金鑰。

### 6.6 工程成本小結（供決策）

方案 D 不是「加個功能」，而是**架構轉向**，主要成本：① 資料層從伺服器搬到裝置端（D1/D2 擇一）、
② 全面改 UUID 主鍵＋修改時間＋墓碑欄位、③ 加密匯出匯入、④ 合併與衝突引擎、⑤ 防遺失 UX。
換得的是新優先序前三項（免審查、防駭、隱私）的**根本性**滿足。建議：**先拍板走方向二（D）後，
再進一步決定 D1 或 D2**，才值得投入上述改造。
