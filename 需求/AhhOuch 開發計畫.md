```
┌─────────────────────────────────────────────┐
│        使用者瀏覽器（電腦版 / 手機版）            │
```

## 3. 技術選型說明

- **語言**：Python 3.11+，符合「pip 套件清單一鍵安裝」「打包單一 exe」。
- **後端**：FastAPI + Uvicorn。輕量、文件完整、好打包；同時提供 REST API 與靜態網頁。
- **前端**：Vue 3（已定案）。功能較多（看板拖曳、多組心得、動態評分表單）時較好組織與維護。
  - 開發期可用 Vite；打包前先 `build` 成靜態檔，交由 FastAPI 提供，整包進 exe。
- **資料庫**：SQLite（搭配 SQLModel／SQLAlchemy）。單一檔案、免安裝、適合單機，置於 `DATA/`。
- **拖曳**：SortableJS，支援同／跨看板拖曳與插入符號提示（C9）。
- **圖片貼上**：瀏覽器 Clipboard API / `paste` 事件接收剪貼簿圖片（詳見 §6 PoC）。
- **發布排版圖片**：前端 `html2canvas` 將卡片/班表 DOM 轉圖，或後端 Pillow 繪製（二擇一，詳見 §6 PoC）。
- **打包**：PyInstaller `--onefile`，輸出單一 exe。

## 4. 系統架構與模組設計

### 建議專案目錄結構

```
AhhOuch_Edit/
├─ run.py                    # 進入點：啟動 FastAPI（打包時的進入點）
├─ app.toml                  # ★ 單點修改：軟體名稱、版次（含打包檔名）(X3)
├─ requirements.txt          # 套件清單 (X4)
├─ install.bat               # 一鍵安裝環境與套件 (X4)
├─ run_dev.bat               # 測試階段啟動指令
├─ build.py                  # 打包腳本（輸出到 打包輸出/、清過程檔）(X1/X2)
├─ README.md                 # 說明文件 (X5)
├─ backend/
│  ├─ main.py                # FastAPI app、靜態檔掛載、啟動建立 DATA/
│  ├─ version.py             # 讀取 app.toml，供 API 與 build 共用 (X3)
│  ├─ config.py              # 路徑常數、預設值、DATA 位置
│  ├─ database.py            # SQLite 連線、初始化、預設資料
│  ├─ models/                # 資料模型（spa/board/card/review/template/schedule…）
│  ├─ routers/
│  │  ├─ customer.py         # 客人 API（養身館/看板/卡片/心得/模板）
│  │  └─ store.py            # 店家 API（資訊卡片/班表/草稿）
│  └─ services/
│     ├─ image_service.py    # 圖片上傳/貼上儲存、封面設定 (S3/C13)
│     ├─ shift_calculator.py # 班表自動換算 1.5h×12h (S10)
│     ├─ publish_service.py  # 發布排版圖片＋純文字產生 (S5/S11/P1)
│     └─ sorting.py          # 卡片 Unicode/手動排序 (C10)
├─ frontend/
│  ├─ desktop/               # 電腦版（優先開發）
│  └─ mobile/                # 手機版（後期，先保留空架構）
├─ DATA/                     # ★ 軟體輸出 (X6)
│  ├─ ahhouch.db             # SQLite 資料庫
│  └─ images/                # 上傳/貼上的圖片
└─ 打包輸出/                 # 打包後的單一 exe (X1)
```

### 4.1 客人 API（`routers/customer.py`）
- 養身館 CRUD：僅需名稱即可建立；含地址、幹部（C1/C2）。
- 看板：增減、改名、排序；建立養身館時自動產生五個預設看板（C5/C6）。
- 卡片：在看板下新增（僅需標題即可建立）、跨/同看板移動、排序（C7/C9/C10/C11）。
- 心得：多組增刪改、套用模板、評分與補充文字、新增時帶入前次內容（C16–C22）。
- 評分模板：多組、命名、項目增刪改、預設模板/項目（C17/C18/C20）。

### 4.2 店家 API（`routers/store.py`）
- 美容師資訊卡片 CRUD：名字、多圖＋封面、完整/簡短介紹（S2–S4）。
- 班表 CRUD：標題、出勤人員選取、各人員出勤時段（S6–S10）。
- 草稿：發布後留存，可載入修改後再發布（S12）。

### 4.3 image_service（圖片，S3/C13）
- 接收「檔案上傳」與「剪貼簿貼上（base64）」兩種來源，存入 `DATA/images/`，DB 僅存路徑。
- 支援多張、設定其一為封面；圖片非必填。
- 卡片在看板僅回傳封面圖縮圖供前端顯示（C8）。

### 4.4 shift_calculator（班表自動換算，S10）

```
輸入：上班開始時間（例 18:00）
規則：每班 1.5 小時，列出涵蓋「開始後 12 小時」的班次
輸出（核取方塊清單，勾選後加入顯示區）：
  18:00、19:30、21:00、22:30、00:00、01:30、03:00、04:30 …
```

> 邊界（是否含第 12 小時整點那班、共 8 或 9 班）列入 §9 待確認。

### 4.5 手動時段輸入（24 時制自動轉換，S9）
- 前端輸入框 + [+] 按鈕；輸入純數字時自動格式化（`1830` → `18:30`），由 `time_utils` 驗證與轉換。

### 4.6 publish_service（發布，S5/S11/P1）
- **第一階段（Phase 4/5）**：同時產生「排版圖片」與「純文字」兩種，供使用者手動貼到群組。
  - 卡片發布：可勾選完整或簡短介紹（S5）。
  - 班表發布格式（S11，依需求範例）：

    ```
    （標題列）
    #199
    按摩力道足夠。特別擅長處理肩頸痠痛
    10:30、12:00、13:30、18:00

    #202
    按摩力道稍小，特別擅長處理大腿痠痛
    12:00、15:00、18:00
    ```
- **第二階段（Phase 6）**：串接 LINE 等官方 API 自動發布（P1，詳見 §6 PoC）。

### 4.7 version.py + 版次設定檔（X3）
- 單一設定檔 `app.toml`（根目錄）：

  ```toml
  # app.toml — 修改此處即可變更軟體名稱與版次（含打包檔名）
  app_name = "AhhOuch"
  version  = "1.0.0"
  ```
- `version.py` 讀取此檔，供前端標題、關於頁、`build.py` 決定輸出檔名（如 `AhhOuch_v1.0.0.exe`）共用。
- **唯一事實來源**：其他檔案一律從此取值，不重複硬編碼。

### 4.8 requirements.txt + install.bat（X4）
- `requirements.txt` 列出 fastapi、uvicorn、sqlmodel、pillow、pyinstaller 等。
- `install.bat`：一行指令 `pip install -r requirements.txt`（README 同步說明）。

### 4.9 README.md（X5）
須包含：安裝指令、測試啟動指令、打包指令、改名改版的檔案（`app.toml`）、打包輸出位置（`打包輸出/`）。

## 6. PoC / 待研究項目

> 不阻擋整體開發，但需在對應階段前先驗證可行性與選定方案。

### 6.1 剪貼簿貼上圖片（S3/C13）— Phase 3 前驗證
- 瀏覽器 `paste` 事件 / Clipboard API 接收剪貼簿圖片並上傳。
- 待驗證：截圖貼上與複製檔案貼上的相容性（見 §9 待確認 2）。

### 6.2 發布排版圖片產生方式（S5/S11）— Phase 4 前選定
- **方案 A（前端）**：`html2canvas` 將卡片/班表 DOM 直接轉成 PNG，所見即所得、實作簡單。
- **方案 B（後端）**：Pillow 依資料繪製圖片，排版穩定、可控，但需自行排版字型（含日文）。
- 初步傾向 **方案 A**；複雜排版或字型問題再評估方案 B。

### 6.3 LINE 等官方 API 自動發布（P1）— Phase 6
- 確認目標平台（LINE？多平台？見 §9 待確認 1）、申請官方帳號與權杖、群組推送方式。

## 8. 風險與注意事項

- **打包流程盡早驗證**：M0 即驗證 PyInstaller 能把 FastAPI＋Vue 靜態檔包成單一 exe，避免後期才發現打包問題。
- **前端打包整合**：Vue 需先 build 成靜態檔再交 FastAPI 提供；需確認靜態資源路徑在 frozen（exe）環境下正確（以執行檔所在目錄為基準建立 `DATA/`）。
- **日文字型**：若採後端 Pillow 產生發布圖片，需內嵌支援日文的字型，否則日文會缺字。
- **圖片儲存體積**：多張圖片存於 `DATA/images/`，需注意空間與備份（見 §9 待確認 3）。
- **多使用者擴充**：初期單機單人，但資料模型/路徑設計應預留 `user` 維度，降低日後改動成本。
- **Unicode 排序**：C10 預設排序須以標準 Unicode 比較，注意中日文與符號的排序一致性。
