# AhhOuch_Manager 幹部系統手機版開發計畫

> 本文件依據 [`AhhOuch 多人使用需求.md`](./AhhOuch%20多人使用需求.md) 與
> [`AhhOuch 需求.md`](./AhhOuch%20需求.md)（店家功能段）撰寫，作為「幹部系統手機版轉型」的開發規劃。
>
> 核心決策：
> - **系統對象**：店家幹部（非消費者），全功能開放，不設付費管控。
> - **目標平台**：手機為主，優先支援 iOS；PWA 方式讓幹部透過 Safari「加入主畫面」使用。
> - **資料存放**：本機優先（Local-First），各幹部資料全存自己手機，互不共享、開發者不持有。
> - **後端**：**不建立開發者伺服器**；App 完全在手機本機運作，班表發布透過手機原生分享功能傳送至 LINE 等社群軟體。
> - **無授權機制**：幹部系統非營利、全功能開放，無序號、無帳號、無金流。
> - **專案定位**：現有 repo（`AhhOuch_Manager`，app.toml 1.7.0／git V2.0.0）已完成「店家→幹部」更名與客人模組剝除（詳見 [`AhhOuch_Manager 開發報告.md`](./AhhOuch_Manager%20開發報告.md) §12 的 11.36），本計畫即為此 repo 的下一步：從「桌面 exe」轉型為「手機 PWA」。
>
> **待確認事項**（§10）中列有數個影響架構的問題，**建議在 M0 前確認**。
>
> 文件版本：v1.0 ｜ 建立日期：2026-06-24

### 與現有版本的關係

現況（見 [`AhhOuch_Manager 開發報告.md`](./AhhOuch_Manager%20開發報告.md)）：現有 repo 為「**Python FastAPI 後端 + Vue 3 前端 + 集中式 SQLite + PyInstaller 打包為桌面 exe**」的架構。原本客人與店家雙角色共存，**2026-06-23 起已執行拆分**：移除客人系統、「店家」全面更名為「幹部」，現行 repo 即為純幹部系統桌面 exe（app.toml 1.7.0／git V2.0.0）。

本計畫目標：將幹部系統從「桌面 exe（需電腦開啟伺服器）」轉型為「**手機 PWA（無需電腦、無需伺服器、各幹部自己手機上獨立運作）**」。

> 注意：現行架構是「FastAPI 後端 + 本機伺服器 + 集中式 SQLite 檔，使用者透過電腦上的網址連線」；本計畫的目標型態是「**無中央伺服器、資料存在每支手機、不需要開發者的電腦**」。兩者的資料層思路相反，前端技術路線與本機儲存方式（§6.1）是本計畫最大的工程量來源，並非單純沿用現有 exe。

### 核心架構流程

```
┌─────────────────────────────────────────────┐
│   幹部手機：AhhOuch_Manager PWA（iOS）          │
│   ┌───────────────────────────────────────┐ │
│   │  幹部系統 UI                            │ │
│   │  ├ 美容師資訊管理（卡片、圖片、文字）        │ │
│   │  └ 班表管理（出勤選人、時段、草稿）          │ │
│   │  本機儲存（業務資料 + 圖片，手機沙盒）        │ │
│   │  全功能開放，無任何付費限制                 │ │
│   └───────────────────────────────────────┘ │
└───────────────┬─────────────────────────────┘
                │ 班表／卡片發布時：呼叫 Web Share API
                │ 傳送格式化文字（或圖片）至 LINE 等
┌───────────────▼─────────────────────────────┐
│   手機原生 LINE / 社群軟體（系統層級）             │
│   由幹部的手機 OS 處理，與 App 無後端關聯          │
└─────────────────────────────────────────────┘

※ 開發者不建立、不維運任何伺服器。
```

## 3. 架構決策摘要（為何如此選）

| 決策面 | 採用方案 | 捨棄方案與原因 |
| --- | --- | --- |
| App 形態 | **PWA（加入主畫面）** | 捨原生 App（需上架 App Store，需開發者帳號，成人內容過審風險高） |
| 後端 | **無開發者伺服器** | 捨保留 FastAPI 伺服器：幹部需要「自己手機獨立跑」，不能依賴開發者電腦開著才能用 |
| 資料存放 | **本機優先（手機沙盒）** | 捨雲端集中儲存：無需開發者維運資料庫，且圖片資料涉及隱私，不宜上雲 |
| 跨裝置 | **手動匯出／匯入備份檔** | 不自建同步伺服器；iOS PWA 本機資料不保證能被 iCloud 原生備份完整涵蓋，故以手動備份保底 |
| 班表發布 | **Web Share API** | 手機 OS 原生分享，可傳送格式化文字至 LINE 等 App；開發者無需維護任何發布端點 |
| 授權／帳號 | **完全無** | 幹部系統非營利，無需任何帳號或授權機制 |

## 4. 系統架構與模組設計

### 4.1 前端 App（PWA，幹部系統）

- 沿用既有幹部模組功能：美容師資訊卡片、班表管理、草稿、發布。
- **PWA 化**：加入 `manifest.json`（名稱、圖示、`display: standalone`、主題色）與 Service Worker（離線快取、可安裝）。iOS 經 Safari「加入主畫面」。
- **本機資料層**：資料與圖片改存手機端（見 §4.2）。
- **無 Feature Gate**：全功能無限制，無免費版／付費版區分。

### 4.2 本機資料層（Local-First，取代集中式 SQLite）

- 業務資料與圖片皆存於各自手機端沙盒：
  - **PWA 純網頁路線（優先評估）**：以 **IndexedDB** 存資料，圖片以 Blob 存 IndexedDB；或採 **SQLite WASM + OPFS** 取得 SQL 介面。
  - **Flutter / React Native 路線（備選）**：直接用裝置 SQLite ＋ 檔案系統沙盒。
- 設計原則：**App 完全離線可用**；無任何時刻需要連接開發者伺服器。
- 各幹部資料完全獨立，不互相影響。

> 技術路線（PWA 純網頁 vs Flutter/RN）為本計畫**首要 PoC**，見 §6.1。

### 4.3 班表發布（Web Share API）

- 班表完成後，App 在本機將內容格式化為指定文字格式（標題、各人編號／簡短介紹／時段）。
- 呼叫 **Web Share API**（`navigator.share({ text: … })`）調起手機的原生分享選單，幹部選擇 LINE 或其他社群 App 傳送。
- 單張美容師卡片亦可透過同樣流程分享（含圖片，見 §6.2 待驗證）。
- 完全不依賴開發者伺服器，發布邏輯在手機端本機處理。

### 4.4 備份與換機（手動匯出／匯入）

- 提供「**匯出備份**」功能：將本機所有資料（美容師卡片、圖片、班表草稿）打包為單一備份檔，存至手機相簿或「檔案」App（或透過分享傳送至 iCloud Drive / Google Drive）。
- 提供「**匯入備份**」功能：讀取備份檔還原至新手機。
- 此設計為換機與防資料遺失的主要保障，無需開發者介入。

## 5. 資料結構設計（初稿）

```

  id / 名稱_or_編號* / 完整介紹 / 簡短介紹 / 圖片清單(Blob refs) / 封面圖id / 排序
```

## 6. PoC / 待研究項目

> 不阻擋整體規劃，但需在對應階段前先驗證並選定方案。

### 6.1 前端技術路線選型 —【最優先，M1 前定案】

- **路線 A：PWA 純網頁**（沿用既有 Vue 前端 + Service Worker + IndexedDB / SQLite WASM）
  - 優點：最大程度沿用現有 Vue 幹部模組、單一程式碼、iOS/Android 通吃。
  - 待驗證：iOS Safari PWA 的本機儲存穩定度、圖片大量存取效能、加入主畫面後的儲存持久性。
- **路線 B：Flutter / React Native**（匯出 PWA ＋ iOS 直接安裝）
  - 優點：原生 SQLite 與檔案沙盒、效能佳。
  - 待驗證：需重寫前端、學習成本、與既有 Vue 程式碼落差。
- 初步傾向 **路線 A（PWA）** 以沿用既有資產；若本機儲存或 iOS 限制踩雷再評估 B。

### 6.2 Web Share API 圖片支援 — M4 前驗證

- `navigator.share({ files: [imageFile] })` 在 iOS Safari 是否能攜帶圖片分享至 LINE。
- 若圖片分享不可行，退而求其次：先分享格式化文字，圖片由幹部另行複製貼上。

### 6.3 iOS PWA 本機儲存持久性 — M1 前驗證

- iOS Safari 對 IndexedDB / OPFS 有容量上限與清除策略；長期不用或清快取可能遺失資料。
- 驗證儲存量（多張圖片）、PWA 離線啟動穩定性、清快取後資料是否保留。
- 若持久性不足，匯出備份功能（§4.4）為必要保障。

### 6.4 手機相簿存取 — M3 前驗證

- PWA 路線下，透過 `<input type="file" accept="image/*">` 可讓幹部從相簿選圖或直接拍照。
- 驗證多張上傳、大圖縮圖、圖片貼上（剪貼簿）是否在 iOS Safari 可用。

### 6.5 備份檔格式 — M6 前定案

- 備份需含結構化資料（JSON）與圖片 Blob；考慮打包為 `.zip` 或自訂格式。
- 驗證 PWA 在 iOS 是否能產生並觸發下載此類檔案，以及重新讀取匯入的可行性。

## 8. 班表發布標準格式

> 出自 [`AhhOuch 需求.md`](./AhhOuch%20需求.md) §店家「班表」，整理為 App 輸出格式規範。

發布格式（由上至下）：

```
{標題列（多行文字，由幹部輸入）}

{出勤人員1的卡片名稱_or_編號}
{出勤人員1的簡短介紹}
{出勤時段1}、{出勤時段2}、…

{出勤人員2的卡片名稱_or_編號}
{出勤人員2的簡短介紹}
{出勤時段1}、{出勤時段2}、…
```

實際範例：

```
本週班表

#199
按摩力道足夠。特別擅長處理肩頸痠痛
10:30、12:00、13:30、18:00

#202
按摩力道稍小，特別擅長處理大腿痠痛
12:00、15:00、18:00
```

### 出勤時段輸入方式

幹部可選擇以下其中一種（或混用）：

1. **手動輸入**：文字框旁有 `[+]` 按鈕，輸入後加入時段列表；採 24 時制，輸入「1830」自動轉換為「18:30」。
2. **自動換算**：輸入上班開始時間，系統依每班 1.5 小時邏輯列出上班開始 12 小時的班次，各班次有核取方塊，勾選後自動加入時段列表。

## 9. 風險與注意事項

- **iOS PWA 儲存持久性**：iOS Safari 對網頁本機儲存有清除策略，長期不用或清快取可能遺失資料；匯出備份（§4.4）為必要保障，需在 §6.3 完成驗證（D4）。
- **Web Share API 相容性**：iOS 15 以上支援 `navigator.share()` 含檔案分享；若舊版 iOS 不支援，需提供降級方案（如複製文字至剪貼簿）。
- **圖片容量**：多位美容師、多張圖片累積後可能超過 IndexedDB 實際可用空間；需在 §6.3 驗證並提示幹部管理圖片量。
- **多幹部資料各自獨立**：幹部 A 更新美容師資訊後，幹部 B 的手機上**不會自動同步**；目前規劃下這是預期行為（見 §10 問題 1）。如需同步，須引入伺服器，與「無開發者伺服器」原則衝突。
- **既有程式碼落差**：現行架構為 FastAPI + Vue + 集中式 SQLite；移植至本機儲存需大幅改動資料層，是本計畫最主要的工程量。

## 10. 待確認事項

1. **多幹部資料同步問題**（⚠️ 影響架構）：若多位幹部各自手機資料獨立，「幹部 A 在自己手機更新美容師卡片，幹部 B 的手機不會自動更新」——這樣的設計是否可接受？
   - 若**可接受**：現行無伺服器架構即可，幹部 A 更新後需手動通知幹部 B 或由幹部 B 重新手動輸入。
   - 若**不可接受**（需要幹部間資料同步）：須引入某種共享機制，例如「班表資料由一位主幹部管理，其他幹部唯讀」或「透過備份檔傳遞」，但若要自動同步則需伺服器，與不建伺服器原則衝突。**請在 M0 前確認。**

2. **Android 支援範圍**：需求文件標註「針對 iOS 系統」，Android 是否完全排除，或預期 PWA 在 Android 上也要可用？

3. **單張美容師卡片的發布格式**：發布時是否隨卡片附上圖片，還是只分享文字（完整介紹或簡短介紹）？（影響 Web Share API 圖片分享的 PoC 優先程度，見 §6.2）

4. **備份觸發時機**：備份是「幹部手動點選匯出」，還是「App 定期自動提醒備份」？

5. **現有桌面 exe 的資料遷移**：若有幹部已在桌面 exe 上建立美容師卡片資料，手機版上線後是否需要提供「從 exe 匯出→手機版匯入」的遷移路徑？

6. **圖片上傳來源**：是否需要支援「直接貼上剪貼簿圖片」（除相簿選取與拍照外）？（見 [`待修改.md`](./待修改.md)）
